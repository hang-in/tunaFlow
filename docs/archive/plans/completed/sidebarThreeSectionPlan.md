# tunaFlow 좌측 사이드바 프로젝트 트리 재설계 계획

- 작성자: OpenAI Codex
- 작성 시각: 2026-03-26 23:10 KST

## 목적

현재 `tunaFlow`의 좌측 `Sidebar`는 프로젝트와 conversation 중심 탐색에는 충분하지만,
하네스 구조가 커질수록 아래 문제가 드러난다.

1. 프로젝트 안의 `main conversation`과 `branch / RT`가 서로 다른 성격인데 한 레벨에서 섞여 보인다
2. 실제 코드베이스 탐색 진입점이 없어, 사용자가 프로젝트 맥락과 파일 맥락을 오가기 어렵다
3. 좌측이 "프로젝트 탐색기"라기보다 conversation 목록에 가깝게 남아 있다
4. 하네스 관점에서 `Architect 대화`, `작업 branch`, `코드베이스`가 한 프로젝트 아래 어떻게 연결되는지 약하다

이 문서는 좌측 사이드바를 하나의 깊은 트리가 아니라,
**프로젝트 기준의 계층형 섹션 탐색기**로 재설계하는 방향과 실제 1차 구현 결과를 정리한다.

## 목표 구조

초기 설계는 아래 3축을 기준으로 시작했다.

- `Chats`
- `Branches & RT`
- `Files`

실제 1차 구현에서는 프로젝트 전환 UX를 더 명확히 하기 위해
아래 4섹션 구조로 구체화되었다.

1. `Projects`
2. `Roundtables`
3. `Branches`
4. `Files`

중요:

- 하나의 깊은 단일 트리로 끝까지 중첩하지 않는다
- 같은 프로젝트를 기준으로 내용만 바뀌는 **계층형 섹션**으로 구성한다
- 좌측은 탐색과 전환 중심, 액션/세부 상태는 중앙과 우측 패널이 담당한다
- `Projects` 섹션 안에 project + main conversations를 함께 두고,
  `Roundtables`, `Branches`, `Files`는 현재 선택된 프로젝트 기준으로 표시한다

## 왜 단일 트리가 아닌가

아래 구조는 비추천이다.

- 프로젝트
  - conversation
    - branch
      - files

문제:

1. 깊이가 깊어져 스캔이 어렵다
2. 채팅 객체와 파일 시스템 객체가 같은 트리 노드로 섞인다
3. branch 수가 늘수록 conversation 목록이 무거워진다
4. 파일 트리까지 붙으면 좌측이 지나치게 길어지고 노이즈가 많아진다

따라서 "3단 구조"는 맞지만,
**하나의 깊은 트리**가 아니라 **3개의 섹션**으로 나누는 것이 좋다.

## 섹션 1. Projects

이 섹션은 프로젝트 전환과 main conversation 진입점을 함께 가진다.

포함:

- 프로젝트 목록
- 현재 프로젝트 이름/상태
- main conversation 목록
- 선택된 conversation 강조
- 프로젝트 추가 액션

원칙:

- 프로젝트 노드는 depth 0
- 현재 선택된 프로젝트 아래에 conversations를 즉시 표시한다
- 비선택 프로젝트는 대화를 미리 로드하지 않고, 선택 시 즉시 전환 + expand 한다
- 여기서의 중심은 `Architect 대화`

### 현재 구현 메모

- 프로젝트 선택은 트리 내 프로젝트 행 클릭으로 전환한다 (별도 드롭다운 불필요)
- `Add project`는 검색 영역 아래에 배치한다
- store가 현재 프로젝트의 conversations만 유지하는 것은 프로젝트 중심 설계 원칙에 따른 의도된 구조다

## 섹션 2. Roundtables

이 섹션은 현재 프로젝트 아래의 RT branch 트리를 보여준다.

포함:

- RT branch
- 하위 RT sub-branch
- 현재 active/open RT 강조

원칙:

- `mode === "roundtable"` 기준으로 일반 branch와 분리한다
- `parentBranchId`를 따라 재귀적으로 계층을 표시할 수 있다
- 좌측은 이동/열기 중심, 구체 액션은 중앙과 drawer가 담당한다

## 섹션 3. Branches

이 섹션은 현재 프로젝트 아래의 일반 branch 트리를 보여준다.

포함:

- 일반 branch
- 하위 branch
- 현재 active/open branch
- 상태 badge (`active`, `adopted` 등)

원칙:

- RT는 이 섹션에 넣지 않는다
- 일반 branch만 별도 트리로 보여 주는 편이 스캔이 쉽다
- branch 액션 전체를 좌측에 몰아넣기보다, 좌측은 이동/열기 중심으로 유지한다

## 섹션 4. Files

이 섹션은 현재 프로젝트 코드베이스의 간단한 파일 탐색기다.

1차 목표:

- 프로젝트 루트 표시
- 주요 폴더 / 최근 파일 / 얕은 파일 트리
- 최소한의 펼침/접힘

주의:

- 전체 IDE 수준의 깊은 파일 탐색기를 처음부터 만들 필요는 없다
- 1차에서는 "최근/주요/얕은" 탐색기여도 충분하다
- 향후 `rawq`, `code-review-graph`, context pack과 연결될 수 있는 구조를 염두에 둔다

## 현재 구조와의 매핑

기존 `Sidebar`가 담당하는 것:

- 프로젝트 목록/선택
- conversation 목록
- 일부 검색/상태

재설계 후:

- 프로젝트와 conversation은 `Projects` 섹션으로 통합
- RT 목록은 `Roundtables` 섹션으로 분리
- 일반 branch 목록은 `Branches` 섹션으로 분리
- 파일 목록은 `Files` 섹션으로 추가

즉 새로운 컴포넌트를 전부 새로 만드는 것보다,
현재 `Sidebar`를 **섹션형 프로젝트 네비게이터**로 재조직하는 것이 핵심이다.

## UX 원칙

### 1. 좌측은 "탐색" 중심이어야 한다

좌측은 액션을 다 때려 넣는 곳이 아니다.

권장:

- 선택/열기/전환 중심
- 가벼운 메타 표시
- 무거운 상세 카드 금지

### 2. 섹션은 항상 접고 펼 수 있어야 한다

특히 `Roundtables`, `Branches`, `Files`는 사용 빈도 차이가 크므로
접이식 섹션이 적합하다.

### 3. Files는 가장 늦게 무거워져야 한다

`Files`는 금방 좌측을 과적재시킨다.

따라서 1차에서는:

- 얕은 트리
- 폴더 1~2단까지만
- 필요 시 최근 파일 중심

으로 시작하는 것이 좋다.

### 4. 현재 프로젝트 기준으로 내용이 바뀌어야 한다

좌측은 전역 브라우저가 아니라
**현재 선택한 프로젝트에 대한 탐색 패널**이어야 한다.

### 5. 검색은 현재 프로젝트 범위만 대상으로 한다

메시지 검색, 파일 검색, rawq code search 모두 현재 선택 프로젝트 내부만 검색한다.
전역 크로스-프로젝트 검색은 현재 제품 원칙에 해당하지 않는다.

## 하네스 관점에서의 의미

이 구조는 하네스 흐름과 잘 맞는다.

- `Projects`
  - Architect와의 주제 대화 진입점
- `Roundtables`
  - 토론/검토용 작업 단위
- `Branches`
  - 구현/파생 작업 단위
- `Files`
  - 코드베이스 맥락

즉 사용자는 좌측에서
"어느 대화인가 → 어느 실행 단위인가 → 어느 코드베이스 파일인가"
를 한 프로젝트 경계 안에서 오갈 수 있다.

## 구현 순서

### Phase 1

- `Sidebar`를 `Projects / Roundtables / Branches / Files` 구조로 재정렬
- `Projects` 아래에 현재 프로젝트의 main conversations 즉시 표시
- `Roundtables`, `Branches`는 `mode` 기준으로 분리
- `Files`는 placeholder가 아니라 최소한의 루트/폴더/얕은 파일 목록부터 시작

### Phase 2

- Files 탐색 강화
- 최근 파일 / 중요 파일 / 확장자 아이콘
- branch와 관련 파일 간의 약한 연결 보조

### Phase 3

- rawq / graph / context pack과 연결
- 최근 touched files
- search / jump to file

## 프로젝트 중심 설계 원칙

tunaFlow는 **프로젝트를 최상위 작업 단위**로 다룬다.

핵심 원칙:

1. store는 **현재 선택된 프로젝트의 데이터만** 유지한다
2. 프로젝트 전환은 "다중 프로젝트 병행 뷰"가 아니라 **컨텍스트 스위치**다
3. 검색, 탐색, 상태 표시는 모두 현재 프로젝트 범위 안에서 동작한다
4. 여러 프로젝트의 conversations를 동시에 펼쳐두고 비교하는 UX는 목표가 아니다

이유:

1. 모든 프로젝트의 데이터를 항상 메모리에 올릴 필요가 없다
2. 프로젝트 수가 늘어도 초기 로드 비용과 상태 동기화 복잡도가 일정하게 유지된다
3. 좌측 사이드바의 목적은 "현재 프로젝트 중심 탐색"이므로 전역 동시 가시성은 필수가 아니다

이 구조는 제한사항이 아니라 **의도된 프로젝트 단위 설계**다.
`conversationsByProject` 같은 전역 다중 프로젝트 캐시는 현재 제품 원칙상 필수가 아니다.

## 주의사항

1. 좌측을 하네스 대시보드로 만들지 말 것
2. branch 액션을 과도하게 좌측에 넣지 말 것
3. 파일 트리를 초기에 너무 깊게 만들지 말 것
4. 현재 `BranchBar`, `ChatObjectTabs`, `workspace panel`과 역할 충돌이 없게 유지할 것

## 현재 판정

좌측 사이드바는 지금보다 더 "프로젝트 네비게이터"에 가까워져야 한다.

현재 구현 결과 기준으로 가장 좋은 1차 방향은:

- `Projects`
- `Roundtables`
- `Branches`
- `Files`

의 4섹션형 구조다.

이 방식은:

- 하네스 흐름을 해치지 않고
- 깊은 단일 트리의 스캔 문제를 피하고
- 향후 rawq/code-review-graph와도 연결하기 쉽다.
