# [P0] 인덱싱 파이프라인 장애 — v32 이후 embedding 복구 실패

## 요약

DB 마이그레이션 v32에서 `conversation_chunks.embedding`을 모두 NULL로 clear했지만, **재인덱싱 경로에 버그가 있어서 대부분의 chunk가 영원히 NULL 상태로 남음**. 결과적으로 Vector 검색이 기능 정지 상태.

### 실측 (2026-04-15)

```
프로젝트별 chunk 수 / embedding 있는 것
  tunaflow:       6711 / 266  (4.0%)
  gemento:         327 /  12  (3.7%)   ← 오늘 retrieval 오염의 근본 원인
  secall:          305 / 274  (89.8%)  ← 정상 (v32 이전부터 활발)
  tunainsight:     201 /   0  (0.0%)
  tunapi:           11 /   0  (0.0%)

문서 chunk (source_type='document'):  6877 / 0 (0.0%)   ← 전부 NULL
대화 chunk (source_type='conversation'): 685 / 552 (80.6%) ← 절반 이상 있음
```

(문서 chunk가 전부 NULL인 건 전체 합 기준. gemento만 204/0.)

## 근본 원인 (3중 버그)

### 버그 1: v32 마이그레이션이 embedding을 clear만 하고 복구 훅 없음

**`src-tauri/src/db/migrations.rs:770-802`**:
```rust
fn apply_v32(conn: &Connection) -> Result<(), AppError> {
    // bge-m3 migration: upgrade vec_chunks from 384dim to 1024dim.
    // ...
    conn.execute("UPDATE conversation_chunks SET embedding = NULL", [])?;
    // ...
}
```

v32는 384dim → 1024dim 업그레이드를 위해 모든 embedding을 NULL로 설정. 주석은 "they'll be re-generated by bge-m3"라고 하지만, **자동 재생성 훅이 구현되지 않음**.

현재 재생성 경로는 (i) 새 대화 후 post-completion 훅, (ii) 수동 `index_conversation_chunks` 호출뿐. 둘 다 **기존 NULL chunks를 백필하지 못함** (버그 2 참조).

### 버그 2: `already_indexed`가 NULL을 포함 → 영구 skip 🔴 (진짜 치명적)

**`src-tauri/src/commands/vector_search/index.rs:306-318`** (`index_chunks_blocking`):
```rust
let already_indexed: std::collections::HashSet<String> = {
    let conn = db.read.lock().map_err(|_| AppError::Lock)?;
    conn.prepare("SELECT root_message_id FROM conversation_chunks WHERE conversation_id = ?1")
        .and_then(|mut s| s.query_map([conversation_id], |r| r.get::<_, String>(0))
            .map(|rows| rows.filter_map(|r| r.ok()).collect()))
        .unwrap_or_default()
};

let new_chunks: Vec<_> = chunks.iter()
    .filter(|(root_id, _, _)| !already_indexed.contains(root_id))
    .collect();
```

`WHERE embedding IS NOT NULL` 조건이 **빠져있음**. NULL인 chunk도 "이미 인덱싱됨"으로 간주되어 `new_chunks`에서 필터됨.

**결과**: v32 후 한 번이라도 chunk가 만들어진 root_message_id는 **영원히 재인덱싱되지 않음**. post-completion이 호출될 때마다 "0 new chunks"만 반환.

동일 버그가 `index_conversation_chunks` (async, line 229-237)에도 존재:
```rust
let mut stmt = conn.prepare(
    "SELECT root_message_id FROM conversation_chunks WHERE conversation_id = ?1"
).unwrap_or_else(|_| conn.prepare("SELECT ''").unwrap());
```

### 버그 3: document chunks는 자동 재인덱싱 트리거 부재

**`src-tauri/src/commands/document_index.rs`**:
- document reindex는 `reindex_documents` 명령 수동 호출 또는 fs watcher에 의존
- **v32 migration 이후 전체 document reindex를 자동으로 트리거하지 않음**
- 결과: document chunk 204개(gemento 기준) 전부 NULL, 복구 불가

## 연관된 부차 이슈 (장애의 증폭 요소)

### 부차 1: Semaphore try_acquire skip

**`persistence.rs:242-255`**:
```rust
let acquired = sem.try_acquire();
if acquired.is_ok() {
    match crate::commands::vector_search::index_chunks_blocking(&db, &conversation_id) {
        ...
    }
} else {
    eprintln!("[post-completion] embed semaphore busy, skipping indexing ... (will catch up next completion)");
}
```

RT 또는 빠른 연속 send 시 semaphore 경합 발생 → skip. 주석에 "다음 completion에서 catch up"이라고 하지만, **버그 2 때문에 실제로 catch up 안 됨** (이미 NULL인 chunk는 `already_indexed`로 skip됨).

### 부차 2: `index_chunks_blocking`의 silent failure

**`index.rs:322-327`**:
```rust
let mut embedded: Vec<(...)> = Vec::new();
for (root_id, kind, text) in &new_chunks {
    if let Ok(v) = crate::agents::embedder::embed_text(text, false) {
        embedded.push(...);
    }
    // Err 케이스는 무시. 로깅도 없음!
}
```

`index_conversation_chunks` (async)는 `eprintln!("[vector] embed failed ...")`가 있지만, blocking 버전에는 없음. bge-m3 로딩 실패나 타임아웃이 나도 **조용히 drop**.

## 수정 후보 (미적용 — 사용자 결정 대기)

### Fix 1: `already_indexed`에 `AND embedding IS NOT NULL` 추가 (핵심)

**`index.rs` 2개 함수**:
```sql
-- 이전
SELECT root_message_id FROM conversation_chunks WHERE conversation_id = ?1

-- 이후
SELECT root_message_id FROM conversation_chunks
WHERE conversation_id = ?1 AND embedding IS NOT NULL
```

**효과**: NULL인 chunk의 root_message_id는 `already_indexed`에 들어가지 않음 → `new_chunks`에 포함 → 다음 post-completion 시 embed 시도 + INSERT. 단 기존 NULL row는 **제거해야** 중복 INSERT 안 됨. 아니면 UPSERT 패턴.

더 깔끔한 수정:
```rust
// 신규 chunk filter 대신 "embedding이 없는 chunk"를 UPDATE
let null_chunks = conn.prepare(
    "SELECT id, root_message_id, kind, text_preview FROM conversation_chunks
     WHERE conversation_id = ?1 AND embedding IS NULL"
)?;
// 각 chunk embed → UPDATE embedding + INSERT vec_chunks
```

### Fix 2: document chunks 재인덱싱 트리거

마이그레이션 v35나 app 시작 hook에서 `source_type='document'` chunk가 NULL인 프로젝트를 감지하면 자동 `reindex_documents` 호출. fs watcher가 이미 있지만 파일이 수정되지 않으면 트리거 안 됨 — 명시적 백필이 필요.

### Fix 3: `index_chunks_blocking` embed 실패 로깅 추가

async 버전과 동일하게 `Err(e) => eprintln!("[vector] embed failed for chunk {}: {:?}", root_id, e)` 추가.

### Fix 4 (선택): Semaphore try_acquire → acquire (await)

빠른 연속 send에서도 skip 없이 큐잉. 단 RT 때 대기 시간 늘어날 수 있음. 트레이드오프 검토 필요.

### Fix 5 (선택): v32 migration에 재인덱싱 훅 추가 (후향적 fix)

v35 마이그레이션으로 "v32 이후 embedding NULL인 프로젝트는 앱 시작 시 백그라운드 재인덱싱 큐에 자동 등록" 추가. 기존 사용자 DB 복구용.

## 복구 우선순위

| # | 수정 | 우선순위 | 효과 |
|---|------|---------|------|
| Fix 1 | `already_indexed` + `IS NOT NULL` | **P0** | 치명적 버그 제거. 신규 completion마다 NULL chunks 복구 시작 |
| Fix 2 | document 재인덱싱 자동 트리거 | **P0** | 문서 검색 복구 (현재 0%) |
| Fix 3 | blocking embed 실패 로깅 | P1 | 진단 가능성 향상 |
| Fix 4 | Semaphore await | P2 | 보조 (Fix 1 하면 필요 감소) |
| Fix 5 | 백필 마이그레이션 | P1 | 기존 사용자 복구 |

## 검증 방법

Fix 1 적용 후:
1. 앱 재시작 + 아무 대화 1회 send
2. 로그 확인: `[vector] indexed N new chunks for ...` 에서 N이 큰 숫자여야 함 (기존 NULL chunks 복구)
3. DB 확인: `SELECT SUM(embedding IS NOT NULL) FROM conversation_chunks WHERE conversation_id = '현재대화'` — 대부분 NOT NULL로 바뀌어야 함
4. 다음 retrieval에서 Vector 결과 3개 이상 반환되는지 확인

Fix 2 적용 후:
1. 앱 시작 시 document reindex 자동 시작 로그 확인
2. `source_type='document'` chunks의 embedding NOT NULL 비율 증가 확인
3. 문서 관련 쿼리에서 document chunks가 retrieval에 포함되는지 확인

## 영향 범위 (베타 차단 여부)

**사용자 체감 기능 저하**:
- Vector 검색이 사실상 기능 정지 → retrieval이 FTS5 단독으로 돌아감
- 문서 검색(Document RAG)이 전혀 작동하지 않음
- 에이전트가 관련성 높은 과거 맥락을 놓침 → 같은 대화방에서 이미 한 논의를 반복 요청하는 상황 발생 가능

**베타 공개 전 이 버그 해결 필수**. README에 "ContextPack + 장기 메모리"를 주요 기능으로 내세우고 있는데 실제로는 장기 메모리 검색이 동작 안 하는 상태.

## 관련 로그 패턴 (진단용)

**정상 동작 시**:
```
[vector] indexed N new chunks for <conv_id> (M already indexed)
  N > 0, M은 과거 누적
```

**버그 증상 시**:
```
(post-completion 훅 돌아도 [vector] 로그가 안 찍힘 — N == 0 이면 로그 생략됨)
[retrieval] Vector: 1 chunks (threshold 0.3)   ← 항상 한 자리수
```
