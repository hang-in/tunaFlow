# Prompt Authoring Policy

상태: 초안
작성: 2026-03-30

## 목적

tunaFlow에서 Claude Code용 실행 프롬프트를 작성할 때, 품질이 약해지지 않도록 공통 기준을 고정한다.

이 문서는:
- Codex가 새 프롬프트를 작성할 때 따르는 기준
- 나중에 tunaFlow 내부 프롬프트 생성 기능에 적용할 기준
을 함께 정의한다.

## 핵심 원칙

1. 프롬프트는 짧기보다 **작업 경계가 명확해야 한다**
2. Claude가 임의 해석하지 않도록 **단계, 제약, 완료 기준**을 준다
3. 실제 코드베이스 기준으로만 작업하게 한다
4. 언어, 출력 형식, 금지사항을 명시한다
5. 필요할 때는 **사용할 skill**까지 지정한다

## 강한 프롬프트의 필수 구성

모든 구현 프롬프트는 가능하면 아래 순서를 따른다.

### 1. 제목

- 작업 이름이 한 줄에 드러나야 한다

### 2. 프로젝트 경로

- 대상 repo 절대 경로

### 3. 먼저 읽을 문서

- 관련 plan/reference/how-to를 명시

### 4. 먼저 확인할 파일

- backend / frontend / store / command 등 실제 진입점을 구체적으로 적는다
- “관련 파일” 수준이 아니라 가능한 한 파일명을 준다

### 5. 작업 시작 전 의견

- Claude가 구현 전에 핵심 판단을 짧게 먼저 말하게 한다
- 방향이 맞는지 빠르게 검증하는 장치다

### 6. 작업 목표

- 이번 단계의 목표를 한 문단으로 고정

### 7. 구현 범위

- 무엇을 해야 하는지 단계/항목으로 쪼갠다
- 가능하면 `1. 2. 3.` 형태로 준다

### 8. 비목표 / 금지사항

- 이번 단계에서 하지 말아야 할 것
- 구조가 새는 것을 막는다

### 9. 검증

- `cargo check`
- `tsc --noEmit`
- 관련 테스트
- 실제 동작 확인 항목

### 10. 출력 형식

- 보고 형식을 고정한다

## 권장 보강 항목

### 언어 고정

필요 시 아래 문구를 넣는다.

- `모든 응답과 보고는 한국어로만 작성하라.`
- `일본어/영어 혼용 금지.`

최근 세션에서 일본어가 섞인 사례가 있었으므로, 한국어 보고가 중요한 작업에는 기본 포함을 권장한다.

### 단계 강제

작업이 크면 아래처럼 단계별로 나눈다.

- `1단계. 현재 구조 분석`
- `2단계. 최소 구현`
- `3단계. 검증 및 문서 반영`

이 구조는 예전 강한 프롬프트에서 품질이 좋았다.

### 완료 기준

단순 “구현하라”보다, 어떤 상태면 이번 단계가 끝난 것인지 적는다.

## Skill 지시 원칙

프롬프트에 skill 관련 지시가 필요한 경우가 있다.

### skill 지시를 넣어야 하는 경우

- 특정 skill 세트가 작업 품질에 직접 영향 줄 때
- frontend / review / OpenAI / Claude / MCP 등 작업 유형별 skill 가이드가 이미 있을 때
- 작업 전에 관련 skill을 먼저 읽는 것이 중요한 경우

### 권장 형식

- `작업 시작 전에 필요한 skill 1~3개만 먼저 읽고 진행하라.`
- `과도한 skill 로딩 금지. 현재 작업과 직접 관련된 것만 사용하라.`

### 주의

- skill 이름 나열만 하고 왜 필요한지 안 쓰면 약하다
- 모든 프롬프트에 무조건 skill을 넣을 필요는 없다
- skill은 **작업 품질을 높이는 최소한의 세트**만 지정한다

## 약한 프롬프트의 징후

아래 중 3개 이상이면 약한 프롬프트일 가능성이 높다.

- 목표는 있는데 단계가 없다
- 금지사항이 없다
- 먼저 볼 파일이 모호하다
- 검증 항목이 약하다
- 출력 형식만 있고 판단 기준이 없다
- 언어/보고 규칙이 없다

## 권장 템플릿

```md
# 작업 제목

프로젝트:
- `/abs/path/to/repo`

먼저 읽을 문서:
- `docs/plans/...`
- `docs/reference/...`

먼저 확인할 파일:
- `src/...`
- `src-tauri/...`

작업 시작 전 짧게 의견을 말하라:
- 왜 이 작업이 지금 필요한지
- 이번 단계에서 무엇을 하지 말아야 하는지

이번 작업 목표:
- ...

구현 범위:
1. ...
2. ...
3. ...

비목표:
- ...
- ...

검증:
- `cargo check`
- `tsc --noEmit`
- ...

출력 형식:
### A. Opinion
### B. Files Changed
### C. ...
### D. Verification
### E. Residual Gaps
```

## tunaFlow 적용 원칙

이 문서는 단순 문서용 규칙이 아니다.

향후 tunaFlow 내부에서:
- 프롬프트를 저장/재사용하거나
- agent별 prompt preset을 제공하거나
- 작업 유형별 prompt template를 자동 생성할 때

이 정책을 기본 작성 규칙으로 사용한다.
