## 1. 멀티 에이전트 오케스트레이션 3대 문제

### 1.1 맥락 붕괴 (41.8%)

에이전트가 자신에게 할당된 작업만 보고, "왜 이 일을 해야 하는지"를 모르는 문제.

- 단일 에이전트 순차 처리를 여러 에이전트가 분담하면 각자 새 세션에서 상황 재파악 → **토큰 최소 10배 증가**
- 목표가 위임 단계를 거치면서 원래 의도에서 벗어나는 **목표 변질**
- 초기 에이전트의 잘못된 가정이 후속 에이전트에게 전파 → **오류 최대 17.2배 증폭**
- 명시적 맥락은 전달 가능하지만, 암묵지(조직 문화, 코드베이스 패턴)는 세션이 끝나면 사라짐

### 1.2 유령 위임 (36.9%)

에이전트가 DONE을 보냈지만 실제로 다음 에이전트에게 인수인계되지 않거나, 특정 상태에서 무한 대기하는 문제.

- 인간 조직에서는 누군가 팔로우업하지만, 에이전트에는 이 복구 로직이 불완전
- Git worktree 기반 병렬 작업도 merge 충돌 시 완료 처리되었으나 결과 없는 경우 발생

### 1.3 검증 오류 (21.3%)

LLM이 자신이 쓴 답을 자신이 채점하므로 "그럴듯한 응답"에 쉽게 완료 처리.

- 실행-검증-판정 3단 구조로 같은 모델도 역할 분리만으로 환각 감소
- 단, 검증 단계마다 토큰 배로 증가 + 속도 저하

## 2. 제안 아키텍처

### 2.1 총괄-워커 패턴

단일 오케스트레이터가 전체 목표를 끝까지 유지하면서 작업을 점진적으로 구체화. 다단계 위임보다 목표 변질을 구조적으로 방지.

> DeepMind 연구: 에이전트 4개를 넘으면 조율 이득이 더 이상 늘지 않음

### 2.2 블랙보드 패턴

에이전트 간 직접 통신(A2A)보다 **공유 환경을 통한 간접 소통**이 더 안정적.

- 이슈 트래커, Git, 메시지 큐 같은 도구가 블랙보드 역할
- 세션 간 상태 유지와 통합된 이슈트래커가 핵심

### 2.3 도메인별 깊은 병렬화

거시적 "자율 조직"보다 **각 도메인 안에서 에이전트에게 적극 위임**하는 접근이 현실적.

- 리서치처럼 병렬화가 자연스러운 영역부터 시작
- 코드 리뷰, 테스트 생성 등으로 확장

## 3. 에이전트 위임 판별 기준 (5가지)

| 기준 | 낮을수록 위임 적합 | 높을수록 인간 영역 |
|------|-------------------|-------------------|
| 오류 비용 | SEO 글, 테스트 데이터 | 고객 대면, 법률, 프로덕션 DB |
| 검증 용이성(역) | 테스트 통과, 빌드, diff | 톤, UX 직관성 |
| 암묵지 의존도 | 명세 기반 버그 수정 | 팀 문화 반영 의사결정 |
| 컨텍스트 범위 | 단일 파일/모듈 | 아키텍처, 크로스팀 |
| 피드백 루프 길이 | 초~분 (코드→테스트) | 주~월 (전략→실행→검증) |

총점 5~10: 에이전트 주도 / 11~18: 하이브리드 / 19~25: 인간 주도

## 4. 토큰 효율성 사례

| 시스템 | 토큰 비율 | 결과 |
|--------|----------|------|
| Gastown | 10x | 생산성 오히려 감소 |
| Paperclip | 5x+ | 인수인계 시 캐시 손실 |
| Anthropic Research | 15x | 토큰이 실제 탐색에 비례 → 가치 있음 |

핵심: 토큰 증가 자체가 문제가 아니라, **토큰이 맥락 재수집에 낭비되는지 vs 실제 작업에 쓰이는지**가 관건.

## 5. tunaFlow 대응 현황

### 맥락 붕괴 대응

| 문제 | tunaFlow 대응 | 구현 |
|------|--------------|------|
| 맥락 손실 | ContextPack — 매 요청마다 전체 맥락 재조립 | `build_normalized_prompt_with_budget()` |
| 목표 변질 | Plan이 SSOT — subtask별 task 파일로 구체화 | `plans` + `plan_subtasks` 테이블 |
| 오류 전파 | 역할별 문서 분리 (architect/developer/reviewer.md) | `docs/agents/*.md` |
| 암묵지 손실 | 주제별 압축 메모리 + 벡터 검색 | `conversation_memory` + `conversation_chunks` |
| 토큰 낭비 | Context mode auto + 버짓 캡 + 사전 추출 | `guardrail.rs` + Insight 파이프라인 |

### 유령 위임 대응

| 문제 | tunaFlow 대응 | 구현 |
|------|--------------|------|
| 무한 대기 | agent_jobs 테이블 + orphan 자동 복구 (2초 폴링) | `RuntimeStatusBar` + `agent_jobs` |
| DONE 미인수 | doom loop 감지 — 3회 실패 시 에스컬레이션 | `plan_events` review_failed 카운터 |
| 상태 유실 | DB가 SSOT — event 유실 시 `list_messages()`로 복구 | SQLite WAL |

### 검증 오류 대응

| 문제 | tunaFlow 대응 | 구현 |
|------|--------------|------|
| 자기 채점 | Developer ≠ Reviewer 역할 분리 | 3-role 파이프라인 |
| 환각 통과 | Review RT (2-agent 교차 검증) | `start_roundtable_run` |
| 기계적 검증 | test_runner + rawq 재스캔 | `run_project_tests` + Auto Fix verify |
| 실패 학습 | failure_lessons FTS5 검색 → rework 주입 | `failure_lessons` 테이블 |

### 설계 원칙 매핑

| 글의 원칙 | tunaFlow 구현 |
|-----------|--------------|
| 총괄-워커 패턴 | Architect가 Plan 유지, Developer/Reviewer는 워커 |
| 에이전트 4개 이하 | 3-role + RT는 2-agent |
| 블랙보드 패턴 | DB(plans/artifacts/findings)가 공유 상태 |
| 도메인별 깊은 병렬화 | Insight 카테고리별 독립 분석, subtask 병렬 그룹 |
| 관측 가능성 | TracePanel + RuntimeStatusBar + Tool Steps |

## 6. 보완 가능 영역

### 6.1 암묵지 구조화
현재 `conversation_memory`가 토픽 요약만 저장. "일하면서 체득하는 패턴"을 프로젝트별 knowledge base로 확장 가능.

### 6.2 위임 판별 자동화
글의 5가지 기준을 Insight의 `fix_difficulty` 평가에 통합. 오류 비용/검증 용이성/컨텍스트 범위를 자동 산출하여 auto/guided/manual 판정 정확도 향상.

### 6.3 계층 깊이 제한
현재 `parent_plan_id`로 후속 Plan 무한 중첩 가능. 글의 권장(3단계 이하)에 맞춰 depth 제한 검토.

### 6.4 인수인계 비용 최적화
같은 프로젝트 내 에이전트 전환 시 ContextPack 캐싱으로 토큰 재수집 최소화. 현재 RT에서 `RtContextCache`로 부분 구현.

## 7. 핵심 인용

> "에이전트 여러 개를 띄워서 회사처럼 돌리면 된다"는 가정은 현재 시점에서 검증되지 않았습니다. 필요한 것은 모든 도메인을 한 번에 뒤집는 혁신이 아니라, 각 영역에서 낮은 점수가 나오는 부분부터 위임의 가치를 하나씩 증명하는 작은 승리들입니다.

tunaFlow의 접근:
- "에이전트가 편해야 결과가 좋다" — 에이전트에게 최적의 맥락을 제공하는 것이 핵심
- 사용자가 도메인 지식과 방향을 결정하고, 에이전트가 그 결정을 최적의 조건에서 실행
- 자율 조직이 아니라, **인간지능 주도형 오케스트레이션**
